## 왜 굳이 '홈서버'일까? 
실무에서 클라우드 환경을 다루다 보면 정말 편하긴 하죠. 복잡한 네트워크 보안이나 하드웨어 관리는 대부분 클라우드 업체에서 관리해주니 실경쓸게 줄어든달까...

문득 "내가 진짜 인프라를 이해하고 있는 걸까, 아니면 그냥 클라우드를 다루는 방법만 아는 걸까?" 라는 의문이 들었습니다.

실제로 업무를 하면서 클라우드의 편리함 뒤에 숨겨진 '물리 계층'에 대한 갈증이 계속 있었고, 가상화 기술이 하드웨어 자원을 어떻게 쪼개서 각 VM에 할당하는지, 물리적인 스위치와 랜선 사이에서 패킷이 실제로 어떻게 흐르는지... 이런 것들은 대시보드만 봐서는 이해가 어렵더군요

그래서 집 한 구석에 나만의 '서버'를 만들어 보기로 했습니다.

### 목표
단순히 서버 한 대 띄우는 게 아니라, 실제 웹 서비스 아키텍처를 그대로 녹여보려고 합니다.

Proxmox 도입: 가상화 레이어를 직접 컨트롤하며 자원 관리해보기

4-Tier 분리: WEB-WAS-DB-Storage를 각각 쪼개서 실제 운영 환경처럼 구성

헛질(?)의 기록: 클라우드에서는 겪을 일 없는 물리 네트워크와 하드웨어 트러블슈팅 경험 쌓기


## 하드웨어 선정

| 항목 | 선택 | 선정 이유 |
|------|------|-----------|
| 모델 | Ryzen 7 5825U 미니PC (베어본) | 8코어 16스레드로 VM 여러 대를 돌리기에 충분하다고 판단. TDP 15W로 부담 적음 |
| CPU | 8C / 16T | Web, WAS, DB, Storage 4개 역할에 코어를 나눠 할당하려면 최소 8코어 필요 |
| RAM | 32GB | 가상화 환경에서 각 VM/LXC에 메모리를 배분하면 16GB는 부족. 32GB가 현실적인 최소선 |
| SSD | 1TB NVMe | OS + VM 디스크 이미지를 모두 SSD에서 운용. HDD는 Storage용으로 별도 구성 예정 |

## 네트워크 장비

| 장비 | 모델 | 역할 |
|------|------|------|
| 공유기 | ipTIME AX3000Q | DHCP 서버, 내부 IP 대역 관리 |
| 스위치 | ipTIME HG25005T1 | 2.5G 유선 연결, 기기 간 고속 통신 |

구성은 아래와 같다.

```
벽면 단자 → 공유기(AX3000Q) → 2.5G 스위치(HG25005T1) → 본체 & 미니PC
```

공유기를 앞에 두는 이유는 DHCP를 공유기가 담당하게 해서 모든 기기를 같은 IP 대역(`192.168.219.x`)으로 통일하기 위함입니다. 스위치만 쓰면 기기마다 대역이 달라져서 통신이 안 되는 문제가 생기기 때문이죠(실제로 스위치를 앞단에 위치시켰다 Proxmox서버가 외부 공인 IP를 직접 점유하려고 시도하다 보니 아이피가 계속 변경되어 통신이 꼬이는 문제를 겪었습니다...)

### 추후 계획: VPN 및 도메인 연동

초기 구축은 내부 네트워크에서만 접근 가능한 구조지만, 이후에는 외부에서도 안전하게 접속할 수 있도록 확장할 예정입니다.

- **VPN (WireGuard)**: 외부에서 홈 네트워크에 접속할 때 WireGuard VPN을 통해 터널링. 포트 하나만 열면 되고, 설정이 간단하면서도 속도가 빠른 장점이 있습니다. 개인 docker프로젝트에 안전하게 접근할 수 있게 하기 위함 입니다.
- **도메인 연동**: 개인 도메인을 구매해서 web서버와 연결하고, SSL 인증서를 적용하여 개방할 예정입니다.

최종적으로는 `도메인 → 공유기 포트 포워딩 → Apache or Nginx(Web LXC) → 내부 서비스` 흐름으로 외부 트래픽을 처리

## LXC와 VM, 뭐가 다른가?

자원 배분 설계를 보기 전에, 이 글에서 계속 등장하는 LXC와 VM의 차이를 알아보겠습니다.

**VM (Virtual Machine)** — 
하드웨어 전체를 가상화. 
각 VM마다 독립된 커널이 올라가기 때문에 완전한 격리가 보장. 
그만큼 메모리와 CPU 오버헤드가 크고, 부팅 시간도 비교적 긴편.

**LXC (Linux Container)** — 
호스트의 커널을 공유하면서 프로세스 수준에서 격리. VM에 비해 메모리 오버헤드가 거의 없고 부팅이 수 초 만에 끝남. 단, 커널을 공유하기 때문에 호스트와 완전히 분리되지는 않음. web서버나 DB처럼 커널 수준 격리가 필수가 아닌 서비스에 적합.

| 비교 항목 | VM | LXC |
|-----------|-----|-----|
| 커널 | 독립 (게스트 커널) | 호스트 커널 공유 |
| 격리 수준 | 높음 (하드웨어 수준) | 중간 (프로세스 수준) |
| 리소스 오버헤드 | 큼 | 작음 |
| 부팅 속도 | 수십 초 ~ 분 | 수 초 |
| 적합한 용도 | Docker, 커널 의존 앱, 보안 격리 | 웹 서버, DB, 경량 서비스 |

Proxmox에서는 이 둘을 한 화면에서 관리할 수 있어서, 역할에 따라 적절히 섞어 쓰는 것이 핵심이다.

## 자원 배분 설계

32GB RAM과 8C/16T CPU를 4개 역할에 나누는 설계를 미리 잡았다.

| 이름 | 타입 | 역할 | CPU | RAM | 비고 |
|------|------|------|-----|-----|------|
| Web | LXC | Apache or Nginx Proxy | 1 Core | 1GB | 외부 트래픽 관문 |
| WAS | VM | Docker + Spring Boot | 6 Cores | 16GB | 메인 개발 환경 |
| DB | LXC | MySQL / PostgreSQL | 2 Cores | 4GB | 데이터 저장소 |
| Storage | VM | NAS / Files | 2 Cores | 4GB | HDD 3개 RAID-Z1 예정 |

**왜 LXC와 VM을 섞어 쓰는가?**

- Nginx, DB처럼 호스트 커널을 공유해도 괜찮은 서비스는 **LXC**가 훨씬 가볍고, 부팅도 빠르고 메모리 오버헤드도 VM보다 적기 때문에 LXC로 구성.
- Docker를 돌리거나 커널 수준 격리가 필요한 WAS, Storage는 **VM**으로 구성.

이렇게 하면 남는 자원이 약 5~7 코어, 7GB RAM 정도 확보되어 호스트 운영과 추가 VM 여유분 확보.
